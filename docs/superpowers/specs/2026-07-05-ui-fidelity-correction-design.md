# CLI 客户端视觉/结构对齐修正 Design

**Goal:** 修正 `apps/cli-gui` 与真实设计稿（`docs/ui/Agent Store.dc.html` 的"CLI 客户端"标签页，非早前研究阶段依赖的转述/截图）之间的具体偏差。本次是直接用 chrome-devtools 打开并点击操作设计稿原文件得到的一手结论，纠正了此前几个计划里基于间接转述做出的错误简化决定。

## 背景

用户明确反馈"整体视觉样式不对"、"所有的 UI 都要对齐"。直接打开 `docs/ui/Agent Store.dc.html` 并点击"CLI 客户端"标签页逐屏操作后，发现的偏差远超色板层面——CSS 变量本身（`--wall`/`--accent`/`--green` 等）与 `apps/cli-gui/src/globals.css` 完全一致，真正的问题是**结构性的**：概览仪表盘的卡片布局、图标系统、"本地代理"作为内置供应商的建模方式、供应商配置的呈现方式，都和已实现的版本有实质差异。

## 逐项偏差记录

### 1. IconRail（图标栏）

**现状（已实现）**：6 个图标——概览(LayoutDashboard)、浏览商店(Compass)、分隔线、全部(LayoutGrid)、供应商(ArrowLeftRight)、技能(Sparkles)、MCP(Boxes)，底部一个纯图标 Settings 按钮。

**设计稿实际**：只有 4 个图标——概览(LayoutDashboard，高亮态用整块圆角紫色背景)、供应商(ArrowLeftRight)、技能(Sparkles)、MCP(Boxes)。**没有独立的"浏览商店"图标，也没有"全部"图标**——点击供应商/技能/MCP 图标本身就直接导航到该分类的浏览列表（`categoryFilter` + `navView: 'browse'` 合并成一次点击，不再是两个独立概念分两步操作）。底部是一个渐变紫色圆形头像按钮（`linear-gradient(135deg, #7c82ff, ...)`），点击打开设置——不是纯线框 gear 图标。

### 2. 概览仪表盘整体结构

**消耗趋势卡片**：
- 现状：标题 + 三栏并排显示（今日/近7天/近30天 各自的请求数+费用），底部一条几乎不可见的细线图表。
- 实际：图标+标题+副标题（"用量数据统计"）的卡片头部；**右上角是一个分段 Tab 控件**（今日 | 近7天 | 近30天），点击切换下方数据，不是三栏并排常显；下方是一个**真实的面积填充折线图**（非细线，紫色描边+半透明紫色填充，图表高度约150px，横跨卡片全宽）；图表下方是**4 个带图标、带背景色调**的统计块横排：总费用（紫色调背景）、总 Tokens（中性背景）、总请求数（绿色调背景）、**模型分布**（紫色调背景，我们完全没有做这个字段）。

**分类计数卡片（供应商/技能/MCP）**：
- 现状：卡片内文字纵向堆叠（标签在上，数字在下）。
- 实际：横向布局——左侧是带背景色的图标徽标（供应商=蓝色调 ArrowLeftRight、技能=绿色调 Sparkles、MCP=橙色调 Boxes）+ 标签文字，数字靠右对齐、大字号。

**本地代理卡片**：
- 现状：标题"本地代理"，单行文字"运行中 · N 个监听配置"。
- 实际：标题是 **"local"**（不是"本地代理"），带图标头像（紫色方块+圆规状图标）；右侧是"运行中"文字+绿色圆点；下方**三个并排的行内统计**：监听地址 / 今日请求 / 成功率（不是一句拼接文字）。此卡片与"可更新"卡片**并排两栏**（local 占主要宽度，可更新占次要宽度），不是各自独占一行纵向堆叠。

**可更新卡片**：
- 现状：标题"可更新"，列表项无图标。
- 实际：标题"可更新" + 计数徽标（橙色圆角矩形数字），右侧有"全部"筛选链接（本次不必实现筛选下拉，但视觉上应有这个元素占位或直接省略筛选交互只保留标题计数即可，忠实度以卡片布局为主）；每个可更新项左侧有图标徽标。

**最近请求卡片**：
- 现状：每行左侧"target · model"，右侧"providerSlug（降级）"。
- 实际：每行左侧是**彩色状态圆点**（绿=成功/红=失败，取决于 statusCode）+ 客户端名称（Claude Code / Codex）+ "model → providerSlug"（降级标记直接跟在 providerSlug 后面，格式是 `providerSlug（降级）`，中间无空格）；右侧是**延迟 + 状态码**，状态码按颜色区分（2xx 绿色、其余红色）。

### 3. "本地代理"的建模方式——结构性重做

**现状（已实现，来自 `2026-07-05-overview-dashboard.md` 计划的主动简化决定）**：`local-relay` 是一个独立的全屏视图（`navView: 'local-relay'`），通过点击概览页的"本地代理"卡片进入，展示配置列表 + 内联编辑，不是抽屉式往返导航。这个简化决定写在该计划的 Global Constraints 里，理由是"数据简单，两屏导航增加成本"——**该判断依据的是转述总结，未直接核对设计稿，现已证实是错误的简化**。

**设计稿实际**：`local` 就是**普通供应商列表里的一个条目**，出现在"供应商"分类浏览视图的"已添加"列表最上方，带一个"内置"徽标。它的行内容是：图标 + "local" + 内置徽标 + "N 个配置 · M 个运行中" + 一个 `+`（新增端口配置）按钮。它的**端口子配置直接以缩进的子行渲染在同一个列表里**（连接线样式，圆点状态指示 + 名称 + 端口号 + `×` 删除按钮），始终展开显示，不需要额外点击展开/收起。

点击 `local` 父行 → 右侧详情面板显示：图标 + "local" + "内置 Provider" 徽标 + "by Agent Store" + 描述文字（"Claude / Codex 指向 local 的某个监听端口，请求经该配置按 Level 顺序转发到上游供应商，失败自动降级。"）+ 一行统计（`127.0.0.1 | N 个配置 | M 个运行中`，运行中数字为绿色）。

点击某个端口子行 → 右侧详情面板显示：面包屑"local / 默认" + 返回箭头，右上角一个真正的**拨动开关**（不是文字按钮）+ "运行中"/"已启用"文字；下方"监听端口"分组，一个可编辑的 `host : port` 输入框（host 部分是灰色只读展示，port 部分是可编辑的高亮蓝色文本），底部一行说明文字提示完整 URL。

### 4. 供应商配置——弹窗 vs 内联面板

**现状（已实现，来自更早一个已合并计划 `2026-07-05-provider-edit-form-rebuild.md`）**：点击"编辑"/"配置"打开一个 Radix Dialog 弹窗（`ProviderEditModal.tsx`），居中浮层展示整个表单。

**设计稿实际**：点击推荐列表项的"配置"按钮，**不弹出任何浮层**——该资源被直接加入"已添加"列表（安装动作内联发生），然后**右侧详情面板本身切换成配置表单**（复用与"选中资源看详情"相同的面板区域，不是叠加层）。表单内容本身（适用客户端图标选择、API 密钥输入、更多设置/高级设置折叠区）与我们已实现的字段基本一致，只是**呈现容器从弹窗变成了详情面板的替代内容**。

## 范围决策

- 上述 4 类偏差中，**第 3 项（本地代理建模）和第 4 项（配置面板容器）是对已合并、已测试功能的结构性推翻**，风险和工作量都明显高于第 1、2 项（纯样式/布局调整）。按下方计划拆分，优先做视觉影响最大、改动风险最低的第 1、2 项，随后处理第 3 项，第 4 项作为独立后续计划处理（不在本次范围内，另开计划）。
- 颜色变量本身（`--wall`/`--accent`/`--green`/`--amber`/`--red` 等）已经完全一致，**不需要改动 `globals.css`**——问题是没有正确使用图标系统和色调背景，不是调色板错误。
- 设计稿用的图标是 lucide 图标库（path 数据与 lucide-react 现有图标一致，如 `ArrowLeftRight`/`Plus`/`X`），`apps/cli-gui` 已依赖 lucide-react，不需要引入新图标库。
- 本次不做"模型分布"背后的真实模型级用量拆分统计（后端目前没有按模型聚合的查询）——先用已有的 `getUsageSummary` 数据里 `model` 字段做一次去重计数（distinct model count），匹配设计稿显示的量级（一个个位数），不做完整的模型用量占比图表。这是本次的 YAGNI 裁定，后续如需精确占比可再扩展。
- "可更新"卡片的"全部"筛选链接本次只做视觉占位（渲染但不接筛选逻辑），因为可更新列表本身条目少，筛选价值低，功能优先级让位于其余更visible的结构修正。

## 不做的事（本次 Plan A 范围外，YAGNI 或另开计划）

- 供应商配置从弹窗改为内联详情面板（第 4 项）——另开计划 `2026-07-05-inline-provider-config-panel.md`。
- 模型级别的真实用量占比统计——占位数字即可，见上文裁定。
- "可更新"卡片的"全部"筛选交互——只做视觉占位。
